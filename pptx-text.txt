TASKS

P1
Phase 1 — TypeScript Foundation
Type safety is the safety net you need before a large-scale refactor
GOAL  ·  Introduce TypeScript across frontend and backend before any structural changes
TASKS
Install typescript, ts-node, tsconfig.json for backend Add @types/node, @types/express, @types/socket.io Convert backend .js files to .ts incrementally Convert frontend .jsx to .tsx Create shared/types/ for socket payload definitions Define API response, room, and game state types Enforce noImplicitAny in tsconfig strict mode Add ts build step to CI pipeline
OUTPUT
Typed backend Typed frontend Shared socket contracts Type-safe refactoring baseline
WHY THIS PHASE EXISTS
TypeScript catches the bugs most likely in a large refactor: wrong payload shapes, undefined properties, mismatched socket event names. Without it, Phases 2–5 are much more dangerous.
--- ppt/slides/slide13.xml
P2
Phase 2 — Modular Monolith Refactor
Turn a chaotic codebase into clean, domain-driven architecture
GOAL  ·  Decompose the God Object into ownable domain modules
TASKS
Create src/ directory structure and migrate all files Create modules/auth/ (controller, service, repository) Create modules/game/ (engine, validator, service) Create modules/matchmaking/ (queue, pairing logic) Create modules/leaderboard/ (aggregation, stats) Create modules/daily-challenge/ (pipeline, scheduler) Split gameSocket.js into game.socket + matchmaking.socket Create infrastructure/ for db, redis, sockets, logger
OUTPUT
Clean domain ownership Modular testable services Separated socket concerns Scalable folder structure
WHY THIS PHASE EXISTS
The God Object means one change can break unrelated flows. Domain modules give each concern a single owner — the foundational requirement before adding Redis, PostgreSQL, or any new feature safely.
--- ppt/slides/slide14.xml
P3
Phase 3 — Security Hardening
Fix trust boundaries — the single most critical production requirement
GOAL  ·  Stop trusting the client for anything security-critical
TASKS
Remove userId from all socket event payloads Add socket auth middleware — verify JWT at connect time Store verified identity in socket.data.userId Move JWT to httpOnly secure cookies Implement refresh token endpoint and rotation logic Add Zod validation on every socket event payload Add Zod validation on every HTTP API route Implement rate limiting on auth endpoints and socket events Fix user._id vs user.id inconsistency across codebase
OUTPUT
Server-derived identity Validated payloads Secure httpOnly auth Rate-limited endpoints
WHY THIS PHASE EXISTS
In any competitive multiplayer system, the client is adversarial. Fake identity in socket payloads can corrupt leaderboards, claim false wins, and hijack rooms. This phase removes that entire attack surface.
--- ppt/slides/slide15.xml
P4
Phase 4 — PostgreSQL + Prisma Migration
Game data is relational — stop using the wrong database
GOAL  ·  Replace MongoDB with PostgreSQL for all durable structured data
TASKS
Provision PostgreSQL via Docker compose Install Prisma and initialise the schema Design tables: users, matches, rooms, moves Design tables: daily_challenges, ratings, leaderboard_entries Add foreign keys and indexes on all query paths Add transactions for match completion + stats updates Migrate existing User and Leaderboard data Update all repositories to use Prisma client Add connection pooling configuration
OUTPUT
Relational data model Transactional integrity Indexed query paths Type-safe Prisma client
WHY THIS PHASE EXISTS
Match history, ratings, leaderboards, and daily puzzles form a naturally relational graph. MongoDB's flexibility was never needed — only its downsides remain: no joins, no transactions. PostgreSQL fixes both.
--- ppt/slides/slide16.xml
P5
Phase 5 — Redis Integration
Move all shared transient state out of process memory
GOAL  ·  Externalise realtime state to enable true horizontal scaling
TASKS
Set up Redis via Docker compose Move matchmaking queue to Redis sorted set Move active room snapshots to Redis hashes Move online presence and heartbeat to Redis Move rate-limit counters to Redis Install and configure @socket.io/redis-adapter Add Redis Pub/Sub for cross-instance event broadcasts Add leaderboard cache with TTL-based invalidation Add game timer synchronisation across instances
OUTPUT
Shared matchmaking queue Distributed room state Cross-instance Socket.IO Cached leaderboard reads
WHY THIS PHASE EXISTS
This is the phase that makes horizontal scaling possible. Without Redis, adding a second server immediately breaks matchmaking. With it, every server instance sees the same queue and the same active rooms.
--- ppt/slides/slide17.xml
P6
Phase 6 — Server-Authoritative Game Engine
The server owns the board. The client only renders and sends actions.
GOAL  ·  Centralise all game truth on the server and structurally prevent cheating
TASKS
Create GameEngine class with full board ownership Server generates, stores, and validates all board state Server rejects invalid moves before applying them Server owns and broadcasts game timer to clients Server detects completion and calculates penalties Frontend becomes a read-only view of server events Add reconnect recovery — restore board and timer state Add forfeit logic on disconnect timeout
OUTPUT
Cheat-proof game state Server-owned timers Reconnect recovery Authoritative outcomes
WHY THIS PHASE EXISTS
If the client owns game state, any player with browser devtools can fake a completed board. A server-authoritative engine makes cheating structurally impossible — there is nothing in client state to manipulate.

TASKS
Layer 1: Conflict hint — highlight invalid placements Layer 2: Candidate hint — show possible values for a cell Layer 3: Elimination hint — explain why candidates are removed Layer 4: Technique hint — name the strategy (naked pair, etc.) Layer 5: Reveal hint — show correct value with max penalty Add penalty scoring per hint layer and difficulty tier Track hint usage per match for analytics and leaderboard Expose hint stats in player profile and daily challenge
OUTPUT
5-layer intelligent hints Penalty-weighted scoring Algorithmic depth Hint usage analytics
WHY THIS PHASE EXISTS
A hint engine that names solving techniques shows algorithmic depth a simple 'show answer' button never could. It doubles as a learning tool — a product story that demonstrates both engineering and product maturity.

--- ppt/slides/slide27.xml
Interview Readiness — Questions You Can Now Answer
These are the exact questions strong engineering interviewers ask about backend projects
Q: Why PostgreSQL over MongoDB?
→ Game data is relational — users, matches, ratings, and moves need joins and transactions. MongoDB was structurally the wrong choice from the start.
Q: How does horizontal scaling work here?
→ Redis holds all shared state. Socket.IO uses the Redis Adapter for cross-instance pub/sub. Any server instance can handle any user's connection.
Q: How do you prevent cheating?
→ Server-authoritative engine. The client sends only moves. The server validates against its canonical board. Nothing in client state can be manipulated.
Q: Why a modular monolith, not microservices?
→ Same internal separation with far lower operational cost. One deployment, one codebase, easier debugging. Microservices add unjustified complexity at this scale.
Q: How does matchmaking work at scale?
→ Players push to a Redis sorted set by ELO. A consumer pops pairs, creates rooms in Redis, and emits match_found via pub/sub.
Q: What breaks first at 10,000 users?
→ Redis throughput becomes the bottleneck. Next: Redis Cluster. PostgreSQL also needs PgBouncer connection pooling at that scale.
After 14 phases, this is no longer a sudoku project — it is a scalable realtime competitive platform.

--- ppt/slides/slide4.xml
Flaws 01 & 02 — Security Failures
Identity is the most critical trust boundary in any competitive multiplayer system
FLAW 01  ·  Client-Trusted Identity
// PublicMatch.jsx
socket.emit('find_match', { userId: user?.id })
// CreatePrivateRoom.jsx  — DIFFERENT FIELD!
socket.emit('create_room',{ userId: user?._id })
Server trusts identity entirely from client payload user?.id  vs  user?._id — already inconsistent across files Any user can send another user's ID and steal wins Leaderboard updates attributed to wrong user silently FIX: derive user from socket.data after server-side JWT verify
FLAW 02  ·  JWT in localStorage
POST /login  →  server returns JWT token
↓
localStorage.setItem('token', jwt)
↓
jwtDecode(token)  →  client sets user state
↓
Client controls its own auth state  ⚠
XSS attack can steal tokens directly from localStorage No token rotation — long-lived exposure window Client decides its own identity — server has no control FIX: httpOnly cookies + access + refresh token flow

Flaws 04 & 05 — Why It Cannot Scale Horizontally
In-memory state is process-local — every instance is completely blind to every other
Load Balancer
↙
↘

waitingPlayer = UserX  ← in process memory

ALL STATE IS LOCAL TO THIS PROCESS Inaccessible to any other server instance
Server Instance B
waitingPlayer = null  ← knows nothing

COMPLETELY BLIND TO INSTANCE A UserY arrives here — will never match UserX
RESULT:  UserX waits on Instance A. UserY connects to Instance B. Each server has its own waitingPlayer — they never pair. Rooms fragment. Reconnects fail. Broadcasts only reach sockets on the same process. MATCHMAKING IS BROKEN.

--- ppt/slides/slide9.xml
Final Tech Stack & Design Decisions
Every choice is a deliberate tradeoff — not a default
LANGUAGE
TypeScript
Type safety catches payload mismatches before runtime — critical during a large-scale refactor
DATABASE
PostgreSQL + Prisma
Game data is relational — users, matches, ratings, moves need joins and transactions
CACHE / QUEUE
Redis
Shared transient state — matchmaking queue, rooms, pub/sub — must live outside the process
ARCHITECTURE
Modular Monolith
One deployment, clean internal domains — same modularity as microservices, far lower operational cost
VALIDATION
Zod
Runtime schema enforcement on every socket event and HTTP endpoint — no silent trust of client input
FRONTEND STATE
Zustand + Context
Feature-local game state in Zustand; cross-cutting auth in Context — no unnecessary re-renders
